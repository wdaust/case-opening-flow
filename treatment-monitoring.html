Treatment Monitoring — Parallel Track

Recurring 30-Day Client Check-In & Scoring Refresh

v4.0 • February 2026

1. Overview

What Treatment Monitoring Is and How It Operates

Key Concept: Treatment Monitoring is a parallel track, NOT a sequential stage. It runs as its own recurring object with an independent clock and does not block any case phase from proceeding.

Core Characteristics

  • 1
    Independent Recurring Object
    Own 30-day recurring clock. Changes to 45 days once treatment is marked complete.
  • 2
    Non-Blocking
    Discovery, Expert, Arb/Med, and Trial all proceed independently. Treatment monitoring never holds a phase gate.
  • 3
    Background Heartbeat
    Functions as a continuous health check on client treatment status, refreshing scores and flagging issues on a recurring basis.
  • 4
    Triggers Per Cycle
    Each cycle fires: client communication appointment, call attempt ladder (if unreachable), scoring refreshes, treatment gap checks.

What Happens at Each Check-In

1
Client Communication Appointment
Day 1–3 of cycle
System creates client check-in task. Para/Legal Asst schedules communication appointment to assess treatment status.
2
Call Attempt Ladder (if unreachable)
Day 3–7 of cycle
If client is unreachable during initial outreach, call attempt ladder triggers with escalating contact methods (call, SMS, email).
3
Scoring Refresh
Day 7–14 of cycle
TSI scoring refresh (5 factors). Client Profile treatment behavior metrics refresh (4 sub-metrics). All scores get updated last_reviewed timestamps.
4
Treatment Gaps Check
Day 7–14 of cycle
System evaluates BJB__Treatment_Gaps_Flag__c. Any gap > 30 days is flagged. Attorney notified within 24 hours if gaps detected.
5
Action Trigger Evaluation
Day 14–28 of cycle
System evaluates all action triggers (demand readiness, treatment gaps, liability scores, provisional status) and fires any applicable tasks.
Clock Change: When treatment is marked complete, the cycle length extends from 30 days → 45 days. All the same checks still fire, but on a longer interval.

Pain & SymptomsRecord current pain levels (1-10), any new symptoms since last check-in, and any improvements. Compare to prior cycle notes.

Provider ComplianceAssess whether client is following provider recommendations: attending PT sessions, taking prescribed medications, following activity restrictions.

6.

8.
Client Concerns & Questions — Address any client worries or case questions
Client ConcernsOpen floor for client questions about case status, timeline, or process. Address concerns and document any action items.

4. Scoring Staleness Rules

How Scores Stay Current — Stale Flags, Audit Trail, Data Trust Meter

Core Rules

Rule Detail Outcome
Last Reviewed Date Every score gets a last_reviewed timestamp updated on each review Enables staleness tracking
90-Day Stale Flag If a score is not reviewed within 90 days, it is automatically flagged as Stale Appears on team lead exception list for immediate attention
“No Change” Checkbox If a reviewer confirms scores remain accurate, they must check a “No Change” confirmation box Creates audit trail. Resets the 90-day clock. Cannot silently skip review.
PROVISIONAL Flag All four scoring systems (Client Profile, Liability, Policy & Collectability, TSI) start as PROVISIONAL Remains PROVISIONAL until Data Hygiene Gates pass for each system

Data Trust Meter

Purpose: The Data Trust Meter rolls up all scoring quality indicators into a single dashboard view per case.

% Required Fields Filled

Tracks how many required Litify fields across all scoring systems have been populated. Missing fields reduce the trust score.

% Scores Current vs. Stale

Ratio of scores reviewed within 90 days vs. those flagged as stale. Goal: 95%+ current across all cases.

Missing Items

Itemized list of what is missing entirely: fields never filled, scores never initiated, hygiene gates incomplete.

PROVISIONAL Blocking: Until Data Hygiene Gates pass, all scores display a PROVISIONAL flag. This flag appears on the Data Trust Meter as incomplete and blocks certain downstream actions (e.g., demand preparation, settlement evaluation).

5. Action Triggers

“What To Do” — Automated Responses Based on Score Thresholds

Condition Threshold System Action Owner
Demand-Ready % < Threshold Below configured demand-readiness percentage System creates demand sprint task set — optional tasks become activated to accelerate case toward demand readiness Auto Para
Treatment Gaps > Threshold Gap exceeds configured days (e.g., 30+ days) Treatment monitoring protocol triggers additional touchpoints — extra client outreach, provider follow-up, gap documentation Auto Para
Liability Score < 40 Liability (SLI/LSI) score below 40 System creates early litigation evaluation task for attorney to assess case viability and strategy Auto Attorney
Score is PROVISIONAL Data Hygiene Gates incomplete Displayed on Data Trust Meter as incomplete. Blocks certain downstream actions (demand prep, settlement eval, final case assessment) Mgmt Para
Note: Action triggers are evaluated during each monitoring cycle (Day 14–28). Multiple triggers can fire simultaneously if multiple thresholds are breached.
